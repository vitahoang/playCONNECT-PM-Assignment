The Problem

01
Siloed Game Pipelines
MyVegas, POP!, MGM and Konami each maintain independent data pipelines and local configs. No shared player identity. No cross-game visibility.
Impact → Zero portfolio-level insight
02
Engineering-Gated Campaigns
Every cross-game campaign requires manual config handoffs and direct engineering intervention. Marketing cannot act without developer time.
Impact → Eng capacity wasted on ops work
03
Slow Time-to-Market
A single cross-game promotion takes 2–3 weeks to launch. Competitors ship campaign updates in hours. Opportunities are missed at every live event window.
Impact → Lost revenue during peak windows
04
No Campaign Visibility
Marketing has no unified view of campaign trigger rates, reward redemptions, or cross-game retention impact. Decisions are made without data.
Impact → No feedback loop to optimise
100%
of cross-game campaigns currently require engineering intervention — zero can be launched by Marketing independently

Platform Alignment

SDK
Event Capture Layer
Utilization
Let clients attach campaign metadata to events (level, purchase, session_start, etc.)
Stream events and their metadata from the game client to the backend
Secure and verify event data to prevent cheating
Technical Assumptions
Can new event types be added via config without a full SDK release?
Is the SDK backward compatible with existing game integrations?
AVA
Player Profiling & Eligibility
Utilization
Evaluate player eligibility by segment, VIP tier, or opt-in status
Return eligibility decision with reason code per campaign
Support live rule updates without campaign downtime
Technical Assumptions
Does AVA expose a synchronous eligibility API with <100ms SLA?
Can eligibility rules be updated on live campaigns without restart?
playOps
Campaign Management Console
Utilization
Build and publish cross-game campaigns without engineering
Monitor live campaign performance via analytics dashboard
Pause, edit, or cancel campaigns in real-time
Technical Assumptions
Can a new Campaign Builder module be added without disrupting the existing console?
Does playOps support role-based access for Marketing vs. LiveOps users?
Coupon
Promo Code Generation
Utilization
Generate unique, player-specific promo codes per campaign trigger
Enforce per-player redemption limits and code expiry rules
Return generated code to Campaign Engine in real-time
Technical Assumptions
Does Coupon expose a REST API for programmatic, per-player code generation?
Can codes be generated in real-time (not batch mode)?
Robinhood
Reward Awarding Engine
Utilization
Award chips/rewards to player account in target game
Support cross-game targeting (trigger ≠ reward game)
Retry failed awards up to 3× with exponential backoff
Technical Assumptions
Does Robinhood support real-time awarding (not batch)?
Can it award to a different game than the trigger source?
Is there an idempotency mechanism to prevent duplicate awards?
NEW
Campaign Engine
—
The Missing Orchestrator
Subscribes to the Event Bus, evaluates active campaign rules, calls AVA for eligibility, triggers Coupon + Robinhood, and logs every action to the Loyalty Ledger. Unifies all 5 existing systems into a single automated flow.

Product Design · Core Epics
07 / 12

Core Epics 1 – 4

EPIC 01
playOps
Campaign Builder — Self-Service UI
Marketing can create, configure, and publish a cross-game campaign without engineering involvement.
Acceptance Criteria
Campaign setup completable in <15 min by a non-technical user
States: Draft → Preview → Scheduled → Live → Paused — all manageable from UI
UI validates required fields and shows estimated eligible audience before publish
Campaigns can be paused or cancelled in real-time without engineering
EPIC 02
SDK
Event Ingestion & Routing
SDK reliably captures standardised in-game events and delivers them to the Campaign Engine within SLA.
Acceptance Criteria
SDK supports configurable custom event types via config — no SDK release required per new event
Events delivered to Event Bus within 500ms, at-least-once delivery guarantee
Event payload includes: player_id, game_id, event_type, value, timestamp
Failed deliveries route to dead-letter queue with alerting; SDK retries up to 3×
EPIC 03
Campaign Engine
Campaign Rule Engine
Automated evaluation of whether an incoming event should trigger a reward — fast, idempotent, and failure-safe.
Acceptance Criteria
Trigger evaluation completes in <200ms p99 under production load
Idempotency enforced per player per campaign — no duplicate rewards on retries
Engine state is recoverable after failure — no events lost during restarts
Supports campaign priority ordering when multiple rules match a single event
EPIC 04
AVA
Player Eligibility
Campaigns respect player segments, VIP tiers, and opt-in status before any reward is issued.
Acceptance Criteria
AVA eligibility API returns a decision within 100ms
Eligibility rules are configurable in playOps and updateable on live campaigns
Ineligible events are logged to Loyalty Ledger but no reward is triggered
Marketing can preview estimated eligible audience size before publishing
Product Design · Core Epics
08 / 12

Core Epics 5 – 8

EPIC 05
Coupon · Robinhood
Reward Delivery
Players receive their reward automatically in the target game within seconds of completing the challenge.
Acceptance Criteria
Reward credited to player's target game account within 2 seconds of trigger evaluation
In-game notification delivered via SDK within 5 seconds of reward generation
Failed deliveries retry up to 3× with exponential backoff; failures alert on-call
All reward events logged: player_id, campaign_id, reward_type, amount, status, timestamp
EPIC 06
Loyalty Ledger
Cross-Game Tracking & Ledger
A unified, queryable audit trail of every campaign event across all games — for CS, BI, and compliance.
Acceptance Criteria
Every trigger, eligibility check, and reward grant is logged with full context
Ledger queryable by BI tooling within 1 hour of event (near-real-time pipeline)
CS can view a player's full campaign history via a read-only console view
Schema versioned; new fields addable without breaking existing consumers
EPIC 07
playOps
Campaign Analytics Dashboard
Marketing has real-time visibility into campaign performance to optimise live campaigns and plan future ones.
Acceptance Criteria
Per-campaign dashboard shows: triggers, eligibility pass rate, rewards granted, redemption rate
Alert fires if error rate >1% or p99 latency >5 seconds
Campaign comparison view for A/B analysis; data refreshed every 5 minutes
Accessible to Marketing and LiveOps without engineering access
EPIC 08
SDK
SDK Modularisation
Game teams can integrate the cross-game event SDK with minimal effort and zero disruption to existing functionality.
Acceptance Criteria
Single trackEvent(type, payload) API — auth, retry, batching handled internally
Integration requires <1 day of engineering effort per game (validated with POP! pilot)
New SDK version is backward compatible — existing AVA/playOps integrations unaffected
Integration guide and changelog published with every SDK release

Project Planning · Risk Management
12 / 12

Risks & Mitigation

Technical
SDK Integration Fragmentation
Probability: High Impact: High
Four game apps with different tech stacks may require per-game customisation for SDK integration, delaying Phase 1 rollout and introducing inconsistent event schemas.
Mitigation
Standardise event schema and SDK contract in Sprint 1 — before any game team integration begins
Run technical spike with POP! in Week 1 to surface incompatibilities early and cheaply
Build SDK adapter layer so per-game differences don't reach the Campaign Engine
Organisational
Game Team Prioritisation Conflict
Probability: Med Impact: High
Each game team has its own roadmap. SDK integration work risks being deprioritised in favour of game features, blocking Phase 2 portfolio rollout.
Mitigation
Secure exec sponsorship and formal commitment from each game lead before Sprint 1 kickoff
Scope integration as a <1-day effort (low ask); embed a dedicated platform engineer per game team
Phase 1 requires only POP! — reduces dependency risk at MVP stage
Operational
Duplicate Reward Triggering
Probability: Med Impact: High
Network retries or race conditions in the Campaign Engine could cause the same player to receive a reward multiple times — creating financial exposure and player trust issues.
Mitigation
Implement idempotency keys at event ingestion — dedup before Campaign Engine evaluation
Add explicit deduplication check in Campaign Engine before calling Coupon or Robinhood
Loyalty Ledger audit log with duplicate detection alerting; CS console shows reward history per player
